# AI Constitution

Vendor-neutral guidance any AI coding assistant (Claude, Copilot, Cursor, etc.) should follow in this project. Covers general development practices and security — not tool mechanics. Tool-specific implementation of this guidance lives in [`.claude/`](../.claude/); framework-specific guidance lives in [`.frameworks/`](../.frameworks/).

"Vendor-neutral" describes this file's _content_ — no rule here assumes Claude-specific mechanics. It doesn't mean every assistant is auto-wired: `.ai/setup.sh` currently only generates a `CLAUDE.md` managed block. A project using another assistant can still point that tool's own config (a `.cursorrules` file, `.github/copilot-instructions.md`, etc.) at this file manually; that wiring isn't generated by this template today.

## Rule Format

Guidance is expressed as discrete rules, not prose. Each rule has:

| Field         | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ID`          | Stable identifier, `<CATEGORY>-NNN` (e.g. `SEC-001`), unique within the table or file it's in — not globally. Every `.frameworks/<name>/rules.md` file uses this same format, so `SEC-001` in this file and `SEC-001` in `.frameworks/dotnet/rules.md` are different rules; qualify by file when it's ambiguous (e.g. two frameworks loaded together, like React + Next.js). No source prefix is needed — each category or framework already has its own table or file, so there's nothing for a bare ID to collide with there. |
| `Title`       | Short human-readable name.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `Description` | One or two sentences stating what the rule requires and why.                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Tags          | Exactly one mutability tag and one enforcement tag.                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |

**Mutability tags** — fixed set, not editable per project:

- `immutable` — cannot be edited or disabled by a consuming project.
- `mutable` — a consuming project may edit or disable it (e.g. to fit local conventions).

**Enforcement tags** — how the AI notifies the user when a rule fires. The built-in three are fixed and not editable, but a project may define additional enforcement tags beyond these if it needs distinct notification behavior:

- `info` — states the rule was applied or not; no action required.
- `warning` — flags a violation and suggests how to resolve it; work continues.
- `stop` — freezes the current step; the violation must be addressed before the AI proceeds.

Rules are grouped into one table per category (`SEC`, `DEV`, `DOC`, ...), matching the ID's prefix (`SEC-001` → `SEC` table). A rule added by a consuming project — rather than pulled in from this template — goes in [`.ai/project-rules.md`](project-rules.md) instead (created on first use by the `add-rule` skill), using a plain `PROJ-NNN` ID with no category segment, so it stays visibly distinct from template-sourced guidance regardless of what it's about, and lives outside any file this template overwrites on sync.

New rules are added with the `add-rule` skill, which picks the right table and keeps ID namespacing and formatting consistent. An ID is never reused once retired — deleting a rule row (a manual edit, or `sync-guidance` letting a template's `immutable` retirement stand) must add the ID to this file's **Retired Rule IDs** list below, and `add-rule` checks that list, not just the visible tables, before assigning a new number.

## Security Rules (SEC)

| ID        | Title                       | Description                                                                                                        | Mutability  | Enforcement |
| --------- | --------------------------- | ------------------------------------------------------------------------------------------------------------------ | ----------- | ----------- |
| `SEC-001` | No Hardcoded Secrets        | Never commit credentials, API keys, tokens, or connection strings; use environment variables or a secrets manager. | `immutable` | `stop`      |
| `SEC-002` | Validate Untrusted Input    | Sanitize and validate all input from users, external APIs, and files before use.                                   | `immutable` | `warning`   |
| `SEC-003` | Confirm Destructive Actions | Get explicit confirmation before irreversible operations (force-push, drop table, `rm -rf`, etc.).                 | `immutable` | `stop`      |
| `SEC-004` | Least Privilege             | Request or grant the minimum permissions needed for a task (file access, network, credentials).                    | `mutable`   | `warning`   |
| `SEC-005` | Dependency Vetting          | Flag newly introduced third-party dependencies for review before adding them.                                      | `mutable`   | `info`      |

## Development Practice Rules (DEV)

| ID        | Title                     | Description                                                                         | Mutability | Enforcement |
| --------- | ------------------------- | ----------------------------------------------------------------------------------- | ---------- | ----------- |
| `DEV-001` | No Silent Failure         | Don't swallow exceptions or errors without logging or surfacing them.               | `mutable`  | `warning`   |
| `DEV-002` | Test Before Claiming Done | Run relevant tests/build before reporting a task complete.                          | `mutable`  | `stop`      |
| `DEV-003` | No Scope Creep            | Don't add unrelated refactors, features, or abstractions beyond what was requested. | `mutable`  | `warning`   |
| `DEV-004` | Small, Reversible Changes | Prefer incremental changes over large rewrites when both accomplish the goal.       | `mutable`  | `info`      |

## Documentation Rules (DOC)

| ID        | Title                   | Description                                                                           | Mutability | Enforcement |
| --------- | ----------------------- | ------------------------------------------------------------------------------------- | ---------- | ----------- |
| `DOC-001` | Explain Non-Obvious Why | Document the reasoning behind non-obvious decisions, not what the code already shows. | `mutable`  | `info`      |

## Retired Rule IDs

IDs that once existed in this file and must never be reused, even though the number is free in the table above (e.g. a `mutable` rule a project removed, or an `immutable` rule the template retired). `add-rule` checks this list, not just the visible tables, before assigning a new ID.

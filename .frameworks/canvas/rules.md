# Canvas API Rules

Framework-specific rules layered on top of [`.ai/constitution.md`](../../.ai/constitution.md). Same rule format and tags — see the constitution for field/tag definitions. Reference: [MDN Canvas API](https://developer.mozilla.org/en-US/docs/Web/API/Canvas_API).

- **ID format:** `<CATEGORY>-NNN` (e.g. `SEC-001`) — unique within this file, no framework prefix needed since this file is already the Canvas API-scoped table.
- **Detected by:** source files calling `getContext('2d'|'webgl'|'webgl2'|'bitmaprenderer')` or containing a `<canvas` element (`*.js`, `*.jsx`, `*.ts`, `*.tsx`, `*.html`)

## Rules

| ID | Title | Description | Mutability | Enforcement |
|---|---|---|---|---|
| `DEV-001` | Wait for the `load` Event Before `drawImage()` | Calling `drawImage()` on an `<img>`/`Image` that hasn't finished loading is a silent no-op — it draws nothing and throws no error, so the bug surfaces as "the canvas is blank" with no exception to point at the cause. Attach a `load` listener (or `await` a promise wrapping it, or use `Promise.all` for multiple images) and only draw once it fires; don't assume `img.src = "..."` makes the image synchronously drawable. | `immutable` | `warning` |
| `DEV-002` | Size the Canvas via `width`/`height` Attributes, Not CSS; Scale by `devicePixelRatio` for Sharpness | The `width`/`height` HTML attributes set the canvas's actual pixel buffer; CSS `width`/`height` only stretches that buffer to fit, distorting the image if the ratio doesn't match the attribute-set size (MDN: "If the CSS sizing doesn't respect the ratio of the initial canvas, it will appear distorted"). Separately, on a high-DPI (e.g. Retina) display, a canvas sized only in CSS pixels renders blurry — set the backing buffer to `cssSize * devicePixelRatio`, scale the context by the same factor, and constrain the CSS size to the intended layout size. | `mutable` | `warning` |
| `DEV-003` | `getContext()`'s Context Type Is Permanent Per Canvas | Once `getContext()` has been called on a `<canvas>` with a given `contextType` (e.g. `"2d"`), a later call on the *same element* with a *different* `contextType` (e.g. `"webgl"`) returns `null` instead of switching — MDN: "It is not possible to get a different drawing context object on a given canvas element." A canvas that needs both 2D and WebGL rendering needs two separate `<canvas>` elements, not one re-contextualized element. | `immutable` | `warning` |
| `DEV-004` | `willReadFrequently` Trades Draw Speed for Cheaper `getImageData()` Reads | Passing `{ willReadFrequently: true }` to `getContext("2d")` forces the browser to use software rendering instead of hardware acceleration — it speeds up and reduces the memory cost of repeated `getImageData()` calls, but at the cost of slower drawing operations overall. Only set it on a canvas that actually does frequent pixel readback (e.g. per-frame pixel manipulation); leaving it on by default or applying it to a purely-drawing canvas trades away acceleration for no benefit. | `mutable` | `info` |
| `DEV-005` | Batch Draw Calls, Use `requestAnimationFrame`, and Pre-render Static Content Offscreen | Per MDN's canvas optimization guide: drive animation loops with `requestAnimationFrame()`, not `setInterval()`/`setTimeout()`, so the browser can sync redraws to the display and pause them in background tabs; round coordinates passed to `drawImage()` with `Math.floor()` to avoid sub-pixel rendering costs; and pre-render content that doesn't change every frame (a static background, a pre-scaled sprite) to an offscreen canvas once, then `drawImage()` that cached result instead of re-issuing the same draw calls on every frame. | `mutable` | `info` |
| `SEC-001` | Cross-Origin Images Without CORS Taint the Canvas — Pixel Readback Throws `SecurityError` | Drawing an image loaded from another origin without CORS approval marks the canvas "tainted": MDN documents that `getImageData()`, `toBlob()`, `toDataURL()`, and `captureStream()` all throw a `SecurityError` on a tainted canvas — this is a deliberate protection against a page using canvas to exfiltrate pixel data from cross-origin images the user didn't intend to expose. To draw a cross-origin image and still read it back, the image must be loaded with `crossorigin="anonymous"` (or `img.crossOrigin = "anonymous"`) *and* the hosting server must send a permissive `Access-Control-Allow-Origin` header — setting the attribute alone does nothing if the server doesn't cooperate. | `immutable` | `warning` |

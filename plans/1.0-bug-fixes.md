## Critical

### 1. Physics accumulator unit mismatch (~100,000 steps/second)

**Where:** `src/index.js:11` (`const dt = 0.01`), `src/index.js:146-151` (`updateTiming`), `src/index.js:123-126` (drain loop).

**What:** `updateTiming` accumulates `performance.now()` deltas, which are **milliseconds**, but the drain loop consumes them in units of `dt = 0.01` — a seconds-scale constant. One 60 Hz frame adds ~16.7 ms and drains ~1,666 sub-steps, so `updatePhysics()` (position updates, full collision sweeps, spawn checks) runs ~100,000×/second — roughly 1,000× more than any sane fixed timestep.

**Impact:** Massive wasted CPU every frame; every downstream physics quirk (bugs 2 and 14) is amplified 1,000×. The game only feels right because entity velocities were implicitly tuned to the wrong unit: with 100 steps/ms, a velocity `dx` works out to exactly `dx` px/ms (bullet `0.5` → 500 px/s).

**Fix:** Keep the accumulator in ms and make the step 10 ms: `const dt = 10`. Because effective speeds are currently `dx` px/ms, `x += dx * dt` with `dt = 10` at 100 Hz produces *identical* movement — no retuning of `dx`/`dy` values needed. The only interaction is `accelerate(dt)`/`decelerate(dt)` in `checkInput`, which currently receive `0.01`; their impulse scale must be preserved (or better, folded into the bug 5 fix by moving input-driven acceleration into the fixed step). Fix together with bugs 2, 3, 8.

### 2. Unpause fast-forwards the world; long pause freezes the game

**Where:** `src/index.js:10` (`lastTick` set at module load), `src/index.js:146-151` (only place it's updated), `src/index.js:290-296` (`pauseGame` doesn't touch it).

**What:** `updateTiming` only runs while unpaused, and nothing resets `lastTick` or `accumulator` when play resumes. The first frame after any unpause computes `elapsed` spanning the entire paused period. This applies to: resuming from pause, the first start after idling on the difficulty menu (`lastTick` dates from module load), and starting a new game after sitting on the game-over screen.

**Impact:** With bug 1, a 30-second pause queues ~3,000,000 `updatePhysics` calls on one frame — a multi-second hang. Even with bug 1 fixed, the world still "catches up": asteroids sweep across the field during the burst and can kill the player instantly on unpause.

**Fix:** In `pauseGame()`, when transitioning to unpaused, set `lastTick = performance.now()` and `accumulator = 0`. Additionally clamp `elapsed` in `updateTiming` (e.g. max 100 ms) as a spiral-of-death guard for slow frames and background-tab throttling.

### 3. Pause debounce is dead — `timeLastPause` never updated

**Where:** `src/index.js:13` (only assignment), `src/index.js:105` and `src/index.js:199` (guards), `src/index.js:290-296` (`pauseGame`).

**What:** `timeLastPause` is set once at page load and never reassigned, so the guards `now - timeLastPause > 1000` / `> 2000` are permanently true after the first two seconds. The `pressed` latch doesn't help against held keys, because OS key auto-repeat re-fires `keydown` and re-latches `pressed[e.code]` every frame.

**Impact:** Holding Enter or P toggles pause every frame (~30 flickers/second). The two guards also disagree (1,000 ms on the paused path vs 2,000 ms on the play path) — whichever window is intended, only one should exist.

**Fix:** Either update `timeLastPause = performance.now()` inside `pauseGame()`, or drop the timestamps entirely and suppress auto-repeat at the source (`if (e.repeat) return;` in the keydown handler), which fixes every held-key edge case at once. One debounce rule, one place.

### 4. Pressing P/Enter on a menu runs a hidden simulation behind it

**Where:** `src/index.js:102-111` (paused branch toggles on Enter/P regardless of `newGame`), `src/index.js:133-137` (menu branch draws over the scene).

**What:** On the difficulty/game-over screens (`paused && newGame`), the paused branch still accepts Enter/P and calls `pauseGame()`. Pressing P sets `paused = false` while `newGame` stays true: from then on every tick runs the full else-branch (timing, score, input, physics, render) and *then* the menu code repaints a black background and menu text over it — the running game is invisible.

**Impact:** On the game-over screen, `score` is still the finished game's score (it's only zeroed on menu confirm), so the hidden game keeps earning +1/second and writes it to the high score — the saved high score inflates with nobody playing. Asteroids also spawn and can "kill" the hidden ship, overwriting `message`. Pressing Enter mostly escapes notice only because `chooseDiff` also consumes the same press and starts a real game.

**Fix:** In the paused branch, ignore the pause toggle while `newGame` is true — menu input belongs to `chooseDiff` alone. This also removes the fragile double-consumption of Enter (pause-toggle + menu-confirm in the same tick).

## Major

### 5. Steering, thrust, score rate, and lifetimes depend on display refresh rate

**Where:** `src/index.js:178-193` (`checkInput`: `rotate(±2)` and `accelerate(dt)` once per rAF frame), `src/index.js:160-175` (frame counter drives score), plus everything keyed to `frame` (bullet lifetime, `frame % 30` spawn gate).

**What:** Entity *movement* is real-time based (ms accumulator), but everything else advances once per `requestAnimationFrame`: rotation is ±2°/frame, thrust is a fixed impulse/frame, and the `frame` counter that drives score-per-second, extra lives, asteroid spawn gating, and bullet expiry counts rAF ticks assuming 60 Hz.

**Impact:** On a 120/144 Hz display the ship turns and accelerates 2–2.4× faster, score accrues 2–2.4× faster (which also inflates high scores and the difficulty ramp, since the asteroid cap is `Math.log(score+1)`), and bullets expire sooner — while movement speed stays correct, making the mismatch obvious.

**Fix:** Move input-driven rotation/thrust into the fixed physics step (query `keys` there), and replace the `frame` counter with real-time accounting (e.g. accumulate ms toward each 1-second score tick; give bullets a spawn timestamp or ms TTL). Note: this overlaps the planned 2.0 frame-rate work, but at 1.0 it is a user-visible correctness bug on any high-refresh display.

### 6. Extra-life award logic is inconsistent

**Where:** `src/index.js:172-175` (time rule: `score % 50 == 0 && frame == 60`), `src/index.js:265-268` (kill rule: `score % 150 <= difficulty.getDiff() - 1`).

**What:** Two unrelated rules award lives. The time rule only fires when the per-second `+1` lands exactly on a multiple of 50 — multiples crossed by kill scoring (`+1..3`) are silently skipped. The kill rule uses a difficulty-widened window on multiples of 150, so on Hard a kill landing on 151 or 152 also awards, and a time-tick award at exactly 150 followed by a kill at 151 double-awards the same threshold.

**Impact:** Lives are granted unpredictably: some 50-point crossings give nothing, some 150-point crossings give two.

**Fix:** One rule: award a life when `Math.floor(score / N)` increases across an update (pick N — 50 and 150 both currently exist; N=100 or N=150 is the usual Asteroids convention). Apply it in one place after any score change.

### 7. Ship hitbox ignores rotation; collision is vertex-vs-AABB only

**Where:** `src/models/spaceship.js:62-76` (`getBounds` returns the unrotated triangle), `src/models/entity.js:61-75` (`checkCollision` tests bound points against the target's axis-aligned box).

**What:** The drawn ship rotates (`getDrawShape` applies `orientation`), but its collision triangle is always nose-up — the hitbox visibly disagrees with the sprite whenever the ship isn't pointing up. Independently, `checkCollision` only asks whether one entity's *vertices* fall inside the other's *bounding box*: an asteroid's box is its maximum vertex radius (up to 2× the base radius), so near-corner passes register hits through empty space, and an asteroid edge can overlap the ship's body without containing any ship vertex and register nothing.

**Impact:** Deaths that look wrong (hit on the sprite's empty side; pass-throughs that should have hit), worse the more the ship rotates.

**Fix:** Rotate the ship's bound points by `orientation` in `getBounds` (same transform the renderer applies). For asteroid tests, prefer a circle test (asteroid center + mean/max radius vs point distance) over the AABB — cheap and much closer to the drawn polygon.

## Minor

### 8. Frame counter cycle is 61 ticks, not 60

**Where:** `src/index.js:161-165`.

**What:** `if (frame <= GAME_FPS) frame += 1; else frame = 1;` lets `frame` reach 61 before wrapping, so the cycle is 61 ticks (verified by simulation). A "second" is ~1.017 s at 60 Hz, and `frame % 30 == 0` gates at both 30 and 60 within one cycle.

**Fix:** `if (frame < GAME_FPS)`. (Superseded if bug 5's fix replaces the counter with real time.)

### 9. Bullet lifetime burns its first tick on the spawn frame

**Where:** `src/index.js:204-213` (bullet created in `checkInput`), `src/index.js:366-373` (`checkDistance` called in `renderGraphics` the same tick), `src/models/bullet.js:53-60`.

**What:** A bullet stores the `frame` value at creation and is supposed to expire after two ticks where the counter re-matches it. But `renderGraphics` runs later in the *same* gameTick with the same `frame` value, so the first of the two allowed matches is consumed at spawn. Effective lifetime is one counter cycle (~1 s), not two.

**Fix:** Give bullets a ms TTL from a spawn timestamp (also resolves the refresh-rate dependence from bug 5). If keeping the counter approach, skip the spawn-frame match. Note the current behavior is pinned by `bullet.test.js` — update those tests with the fix.

**Implemented as:** a ms TTL was applied first, then superseded — bullets no longer wrap at the screen edges (unlike other entities) and instead expire via `Bullet#isOffScreen`, a purely positional check. This still resolves the refresh-rate dependence and avoids a clock-based edge case where a bullet fired just before a long pause would already be time-expired on unpause.

### 10. Confirming the menu with Space immediately fires a bullet

**Where:** `src/index.js:396-401` (menu confirm on `pressed["Space"]`), `src/index.js:205-213` (shooting on `keys["Space"]`).

**What:** Space both confirms the menu and is the fire key. The confirm consumes the `pressed` edge, but `keys["Space"]` is still held on the first playing frame, and `timeLastShoot` is 0, so a shot + sound fire the instant the game starts.

**Fix:** Require a fresh Space press to shoot (track it via a `pressed`-style edge rather than `keys`), or clear `keys["Space"]` when the menu confirms.

### 11. Keys stick when the window loses focus

**Where:** `src/index.js:15-29`.

**What:** `keys` is only cleared by `keyup`. Alt-Tab (or any focus loss) while holding a key means the `keyup` never arrives; the ship keeps rotating/thrusting/shooting until the same key is pressed and released again.

**Fix:** `window.addEventListener("blur", () => { keys = {}; })` (and optionally on `visibilitychange`).

### 12. Reset-high-score dialog leaks keys into the game; `preventDefault` breaks dialog keyboard use

**Where:** `src/index.js:20-29` (global listeners with unconditional `preventDefault`), `src/index.js:46-63` (dialog wiring).

**What:** The window-level key listeners keep running while the modal dialog is open, so Enter/Space/arrows pressed in the dialog also drive the game behind it (confirm the menu, shoot, change difficulty). Worse, `e.preventDefault()` on every keydown suppresses native button activation (Enter activates on keydown; Space's activation is cancelled by keydown default-prevention), making the dialog's Yes/Cancel buttons keyboard-inoperable — an A11Y-004 violation. Blanket `preventDefault` also eats browser keys like F5 and Tab page-wide.

**Fix:** In both key handlers, return early when `resetHighDialog.open` (or when `e.target` isn't the body/canvas). Restrict `preventDefault` to the keys the game actually uses (arrows, Space) so Tab/F5/etc. keep working.

### 13. Asteroids can spawn with near-zero velocity and hug the screen edge

**Where:** `src/index.js:298-351` (`spawnAsteroid`).

**What:** Both velocity components come from `Math.random() * 0.2` (with a random sign on the tangential one), so both can be ~0. Such an asteroid sits half-visible on its spawn edge indefinitely, contributing to the on-screen cap while posing no threat.

**Fix:** Enforce a minimum speed on the into-screen component, e.g. `0.05 + Math.random() * 0.15`.

### 14. Asteroids spawn in a burst up to the cap within a single frame

**Where:** `src/index.js:216-221` (spawn check inside `updatePhysics`, which runs once per accumulator sub-step).

**What:** The spawn condition (`asteroids.length < Math.log(score + 1) && frame % 30 == 0`) is evaluated every physics sub-step, but `frame` is constant for the whole gameTick. On a gating frame, every sub-step below the cap spawns — the field jumps from N to the cap instantly instead of trickling in.

**Fix:** Decide spawns once per gameTick (move the check out of the sub-step loop), or track the last spawn time in ms. Largely masked once bug 1 reduces sub-steps, but the check still doesn't belong inside the drain loop.
